!!! danger "가장 중요한 조건"
    - 지금 보고 있는 PDF에만 맞는 코드로 만들면 안 됩니다
    - 다른 대학 모집요강 PDF에도 쓸 수 있어야 합니다
    - 페이지 번호 고정이 아니라, 대제목, 소제목, 표 의미 같은 **구조적 신호**를 기준으로 동작해야 합니다

## 2-1. 전체 설계 먼저 요청하기

!!! example "설계 요청 프롬프트"
    ```text
    이제 본격적으로 파싱 도구를 만들어줘. 중요한 조건이 있어:

    1. 이 도구는 서울대 모집요강만이 아니라, 다른 대학 모집요강 PDF에도 
       사용할 거야. 지금 보고 있는 PDF에만 동작하는 방식은 피해줘.
       다만 대학마다 형식이 다르니, 공통 파이프라인은 일반화하고
       학교별로 미세조정이 필요한 지점은 따로 드러나게 설계해줘.

    2. 이런 순서로 만들어줘:
       - 먼저 PDF 전체를 훑어서, 우리가 뽑으려는 정보가 어느 페이지에 
         있는지 자동으로 찾기
       - 찾은 페이지들에서 표와 텍스트 데이터 추출하기
       - 추출한 데이터를 정해진 컬럼 형식에 맞게 정리하기
       - Excel 파일로 저장하기

    3. PDF에서 텍스트가 거의 추출되지 않으면 이미지형 PDF일 가능성도 있으니,
       OCR이나 Vision API 같은 대체 경로를 어떤 식으로 붙일 수 있을지도 
       설계에 포함해줘.

    먼저 전체 설계를 보여주고, 내가 확인한 다음에 코드를 작성해줘.
    ```

!!! success ""
    설계를 확인한 뒤에는 짧게 이렇게 답하면 됩니다:
    **"좋아, 그 방식으로 진행해줘. 도구 코드 다 만든 뒤에 작동 원리를 간단히 설명해줘. 실제로 아직 돌리지는 마."**

## 2-2. AI 교차 검증을 개발 루프에 넣기

!!! example "검증 및 실행 프롬프트"
    ```text
    서울대만 스크립트를 실행해서 결과를 보여줘. 그리고 원본 PDF 파일을 
    직접 열어서 봐. 추출된 결과가 PDF 원본과 맞는지 네가 직접 비교해봐. 
    틀린 부분이 있으면 알려줘.
    특히 페이지 걸쳐있는 표와 셀 병합 부분이 제대로 처리됐는지 중점적으로 봐줘.
    빈 칸이 있는데 원래 데이터가 있어야 하는 곳은 없는지, 
    숫자가 들어가야 할 곳에 이상한 글자가 들어간 건 없는지, 
    행 수가 원본 PDF의 데이터 수와 맞는지.
    ```

!!! danger "왜 파싱이 안 되는지 모르겠으면, 이렇게 대놓고 시키세요"
    ```text
    왜 결과가 틀리는지 추측하지 말고,
    원본 PDF를 페이지별로 직접 열어서 추출 결과와 나란히 비교해줘.
    어느 페이지에서 어떤 표가 누락됐는지,
    어느 컬럼에서 어떤 규칙이 빠졌는지 눈으로 확인해서 설명해줘.
    ```

!!! warning "이런 일이 생길 수 있다"

    **시나리오 1: AI가 "일반화했다"고 하는데 페이지 번호가 하드코딩돼 있다**
    ```
    코드에서 "page 3", "page 7" 같은 고정 숫자가 보여.
    이건 서울대에만 맞는 거잖아. 
    페이지 번호 대신 구조적으로 찾는 방식으로 바꿔줘.
    ```
    
    **시나리오 2: 스크립트가 10분 넘게 돌아간다**
    ```
    너무 오래 걸리는 것 같아. 
    지금 어디까지 진행됐는지 알려주고, 
    진행률을 볼 수 있게 해줘.
    ```
    
    **시나리오 3: 결과 Excel이 이전보다 오히려 나빠졌다**
    ```
    아까 Stage 1에서 만든 결과보다 오히려 빈칸이 많아졌어.
    뭐가 달라졌는지 비교해서 보여줘.
    ```
