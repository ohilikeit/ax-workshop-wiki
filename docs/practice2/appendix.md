# 부록. 피드백 방법과 에러 대처

**← 메인으로 돌아가기:** [실습 2 개요](index.md)

!!! danger "실습 중 빨간 글씨 에러가 나와도 정상입니다."
    직접 해석하려고 애쓰기보다, 에러 메시지를 그대로 복사해서 AI에게 전달하면 됩니다.

---

## 에러가 났을 때

!!! quote "이 문장을 그대로 사용하세요"
    ```text
    이런 에러가 났어:
    [에러 메시지를 그대로 복사해서 붙여넣기]

    원인이 뭘지 설명해주고 해결해줘.
    ```

!!! success ""
    핵심은 에러를 이해하는 것이 아니라, **있는 그대로 전달하는 것**입니다. AI가 원인과 해결책을 정리해주면 그 흐름을 따라가면 됩니다.

---

## 이상할 때 피드백하는 법

어떤 단계에서든 이런 식으로 말하면 됩니다. 코드를 읽을 필요 없습니다.

| 상황 | 이렇게 말해보세요 |
|------|-------------------|
| 결과가 너무 많을 때 | "문제가 아닌 것까지 잡히는 것 같아. 5쪽은 목차인데 통과됐어." |
| 결과가 너무 적을 때 | "12쪽에 분명 문제가 있는데 하나도 안 잡혔어." |
| 잘림이 이상할 때 | "03번 문제 사진에 04번 첫 줄이 같이 잘렸어." |
| 왼쪽/오른쪽이 섞였을 때 | "왼쪽 칸 문제인데 오른쪽 칸까지 포함돼 있어." |
| 잘 모르겠을 때 | "결과를 봤는데 맞는지 모르겠어. 정상적인 결과가 어떤 건지 예시를 보여줘." |
| 다시 하고 싶을 때 | "이 단계 처음부터 다시 해줘. 이전 결과는 지우지 말고 새 폴더에." |

## 프롬프트 작성 팁

| 팁 | 나쁜 예 | 좋은 예 |
|----|---------|---------|
| 구체적으로 말하기 | 잘 해줘 | 9쪽에서 문제 번호를 찾아서 한 문제씩 잘라줘 |
| 내 상황 알려주기 | (생략) | 나는 프로그래밍을 모르는 비개발자야 |
| 한 번에 하나씩 | 설계, 구현, 검증을 한 번에 요청 | 먼저 설계를 보여주고 → 확인 후 구현 → 검증 |
| 업무 맥락 주기 | 표를 읽어줘 | 이 표는 세로로 병합돼 있어서 아래 줄도 같은 값이야 |
| 재사용 의식하기 | 지금 PDF만 되면 돼 | 다른 교재에도 쓸 수 있게 만들어줘 |

## 자주 쓰는 개념

!!! example "관찰담"
    내가 문제집을 넘겨보면서 발견한 규칙을 자연어로 AI에게 전달하는 것. 예: "문제 번호는 본문 글자보다 크다"

!!! example "검증 루프"
    결과 확인 → 오류 수정 → 재검증을 반복하는 것

!!! example "디버그 사진"
    AI가 찾아낸 결과를 원본 사진 위에 시각화해서, 왜 틀렸는지 한눈에 파악할 수 있게 만든 이미지

!!! example "Skill"
    자주 하는 작업을 "명령어 하나"로 재실행할 수 있게 만든 재사용 포장

## 매 단계 끝 AI 보고 형식

AI가 장황하게 설명하면 "보고 형식대로 짧게 정리해줘"라고 말하면 됩니다.

```text
1. 한 일: (뭘 했는지 1~2문장)
2. 실행 명령: (뭘 실행했는지)
3. 생성/수정 파일: (새로 생긴 파일들)
4. 확인할 결과물: (내가 열어봐야 할 파일)
5. 다음 단계: (넘어가도 되는지)
```
