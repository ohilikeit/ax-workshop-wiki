# 2. 이미지 공고에서 정보 뽑아 양식 채우기

<div class="stage-nav" markdown>
**← 이전:** [1. 결과를 담을 빈 양식 만들기](stage1.md) &nbsp; | &nbsp; **다음 →** [3. 웹 화면 붙이고 직접 써보기](stage3.md)
</div>

> 1번에서 만든 빈 양식에 채울 데이터를 뽑는 단계입니다. 이미지 공고를 받아서 (1) 너무 길면 알맞게 잘라내고, (2) AI에 보내 텍스트를 뽑고, (3) 한 공고에 여러 포지션이 섞여있으면 자동으로 분리해서, (4) 1번에서 만든 빈 양식에 채워 결과 폴더에 저장합니다.


!!! abstract "이 단계의 목적"
    - 긴 이미지를 안전한 위치에서 여러 조각으로 나누는 모듈 만들기
    - AI에 보내 캐치 폼 양식대로 정리된 결과 받기
    - 한 공고에 여러 사업부문·포지션이 있으면 자동으로 분리
    - 1번의 빈 양식에 채워 결과 폴더에 저장

!!! tip "이번엔 모듈을 세 갈래로 나눕니다"
    하나의 큰 함수에 다 몰아넣지 말고 세 갈래로 나눠서 만들도록 부탁합니다.

    1. **자르기** — 긴 이미지를 안전한 위치에서 여러 조각으로 나누기
    2. **추출** — 조각들을 AI에 보내 카탈로그·스키마에 맞춘 결과 받기
    3. **저장** — 결과를 1번의 빈 양식에 채워 결과 폴더에 저장

    이렇게 나눠두면 나중에 한 부분만 바꿔 끼우기 쉽습니다.

!!! info "이 단계에서 다룰 공고 4건"
    메인 샘플은 한화 공채(긴 이미지 + 멀티 포지션)입니다. 모듈이 완성되면 중간에 다른 3건도 같은 모듈로 돌려봅니다. 공고 형태가 달라도 같은 모듈이 잘 동작하는지 확인하는 흐름입니다.

    | 파일 | 크기 | 특징 |
    |---|---|---|
    | `data/hanwha_2026.jpg` | 1000 × 4038 | **메인 샘플.** 긴 이미지, 사업부문 3개 |
    | `data/hanmi_2026.jpg` | 900 × 8667 | 매우 긴 이미지, 여러 조각으로 분할 필요 |
    | `data/dongaoshuca_2026.png` | 973 × 3531 | 중간 길이 |
    | `data/codeit_2026.png` | 785 × 1425 | 짧은 이미지, 단일 포지션 |

---

## 2-1. 긴 이미지 자르기 모듈

채용 공고 이미지는 세로로 길게 캡처된 경우가 많습니다 (한화 공고는 4000px, 한미 공고는 8600px). 이렇게 긴 이미지를 한 번에 AI에 보내면 글씨가 뭉개져서 인식이 잘 안 됩니다. 그래서 **글자나 표가 잘리지 않는 안전한 여백 위치에서** 여러 조각으로 나눠야 합니다.

!!! info "왜 이렇게 구체적으로 말해주나요?"
    이미지 자르기는 **"왜 잘라야 하는지"와 "어디서 잘라야 하는지"** 두 가지 관찰이 결과 품질을 좌우합니다. 이 두 가지를 내 말로 풀어 전달하는 게 실습 1·2부터 계속 쓰고 있는 **관찰담**입니다. 본인 업무에서도 "왜 이 작업이 필요하고, 어떤 기준으로 해야 하는지"를 설명해주면 AI가 알아서 코드로 바꿔줍니다.

!!! quote "AI에게 이렇게 말해보세요 — 이미지 자르기 모듈"
    ```text
    세로로 긴 채용공고 이미지를 AI에 한 번에 보내면 글씨가 뭉개져서
    잘 못 읽어. 그래서 AI에 보내기 전에 이미지를 여러 조각으로
    미리 잘라두는 모듈을 먼저 만들고 싶어.
    위치는 backend/modules/image_slicer.py.

    [얼만큼 잘라야 하는지]
    - Gemini는 이미지 한쪽 길이가 3000픽셀을 넘으면 강제로 축소해서
      글자가 뭉개져. 그래서 각 조각의 세로 길이가 2500픽셀 안쪽이 되도록
      잘라줘.
    - 이미지 세로가 이미 2500픽셀 이하면 자르지 말고 원본 한 장 그대로 써.

    [자르는 위치 — 글자나 표가 중간에서 끊기면 안 돼]
    - 사람 눈으로 봤을 때 글 줄 사이의 흰 여백이 있는 곳에서만 잘라야 해.
      글자나 표 한가운데를 가로지르는 건 절대 안 돼.
    - "여기쯤에서 잘라야겠다" 싶은 지점 근처에서 가장 가까운 흰 여백을
      찾아서 거기서 자르는 방식으로 해줘.

    [결과물]
    - 잘린 조각들은 지정한 폴더에 chunk_1.png, chunk_2.png ... 순서로 저장.
    - 이 모듈은 앞으로 다른 공고에서도 계속 불러서 쓸 거라서,
      이미지 경로와 저장 폴더만 넘기면 돌아가는 형태로 만들어줘.

    다 만들면 data/hanwha_2026.jpg 로 한 번 직접 돌려서
    output/hanwha_2026/chunks/ 폴더에 조각들이 만들어졌는지 확인하고
    파일 위치를 알려줘. 내가 조각을 하나씩 열어서 글자나 표가 중간에서
    잘린 게 없는지 직접 확인해볼 거야.
    ```

!!! success "이런 결과가 보이면 정상입니다"
    - `output/hanwha_2026/chunks/` 폴더에 `chunk_1.png`, `chunk_2.png` ... 가 생겼습니다
    - 각 조각을 열어보면 글자나 표가 중간에서 잘리지 않고 깔끔하게 나뉘어 있습니다

!!! tip "이상하면?"
    - "조각에서 글자가 중간에 잘렸어" → "글 줄 사이의 흰 여백을 찾는 범위를 더 넓혀봐. 잘라도 되는 위치를 더 보수적으로 판단해줘"
    - "조각 수가 너무 많아" → "각 조각의 세로 길이를 좀 더 키워도 돼. 2700픽셀 정도로 시도해줘"

## 2-2. AI 호출 + 결과 분리 + 양식에 채우기

자른 조각들을 AI에 보내 텍스트를 뽑고, 한 공고에 여러 사업부문·포지션이 섞여있으면 자동으로 분리한 뒤, 1번에서 만든 빈 양식에 채워 저장합니다.

!!! info "왜 사업부문별로 분리해야 해요?"
    한화 공채처럼 한 공고 안에 여러 직군(본사영업·본사지원·디지털금융 등)이 섞여 있는 경우가 많습니다. 캐치 폼은 모집부문 단위로 등록하기 때문에, 결과도 사업부문별로 분리돼있어야 그대로 옮겨 입력할 수 있습니다.

!!! quote "AI에게 이렇게 말해보세요 — 추출 + 분리 + 저장"
    ```text
    @backend/catalog.md @backend/schema.py
    @backend/templates/ @backend/modules/image_slicer.py 모두 참고해줘.

    이미지 공고를 받아서 캐치 폼 양식에 맞게 정리해주는 모듈을 만들어줘.

    1. 이미지를 받으면 먼저 아까 만든 image_slicer 로 잘라.
    2. 잘린 조각들을 모두 한 번에 Gemini에 보내서 카탈로그·스키마에 맞춘 결과를 받아.
    3. 한 공고에 여러 사업부문(예: 본사영업, 본사지원, 디지털금융)이나
       여러 직무(예: IB, Trading, IT)가 섞여 있으면 자동으로 각각 분리해서 정리해.
       포지션이 하나뿐이면 하나만 만들어. 억지로 나누지 말고.
    4. 카탈로그에 없는 정보는 따로 빼두는 칸으로 흘려줘 (그냥 버리지 말고).
    5. 결과는 1번에서 만든 빈 엑셀과 HTML 템플릿에 채워서
       output/<공고이름>/ 폴더에 저장해. 엑셀과 HTML 두 파일 다.

    내가 직접 돌려볼 수 있게 만들어줘.
    이미지 파일 경로 하나만 넘기면 자르기부터 저장까지 다 돌아가는 형태로.

    다 끝나면 마지막에 결과 파일이 어디 있는지 경로를 알려줘.
    탐색기로 직접 열어서 확인할 거야.

    테스트는 data/hanwha_2026.jpg 로 해봐.
    한화 공고는 사업부문이 3개(본사영업/본사지원/디지털금융)니까
    결과에 3개로 잘 나뉘는지 보여줘.
    ```

!!! success "이런 결과가 보이면 정상입니다"
    - `output/hanwha_2026/` 안에 `result.xlsx`, `result.html` 이 생겼습니다
    - 엑셀을 열면 모집부문이 사업부문(본사영업·본사지원·디지털금융) 단위로 3개 행으로 나뉘어 있습니다
    - HTML을 브라우저로 열면 본사영업·본사지원·디지털금융 카드 3개가 보입니다
    - 카탈로그에 없는 항목(예: AI면접, 코딩테스트)이 따로 빼둔 칸에 들어가 있습니다

!!! tip "이상하면?"
    - "모집부문이 분리 안 되고 하나로 합쳐졌어" → 프롬프트에 "한 공고에 사업부문이 여러 개면 반드시 별도로 분리. 한화 공고처럼 직군 3개가 있으면 결과도 3개의 모집부문으로 나뉘어야 해"를 한 번 더 강조
    - "결과가 너무 짧아" → "모든 항목을 빠짐없이 채워줘. 정보가 없으면 비워두지 말고 null로 표시해줘"
    - "결과 파일 위치를 안 알려줘" → "처리가 끝나면 마지막에 결과 파일들의 절대 경로를 한 줄씩 보여줘"

!!! warning "이런 일이 생길 수 있다"

    **시나리오 1: 결과 형식이 매번 다르게 나온다**
    ```
    AI가 같은 이미지를 줘도 결과 형식이 매번 달라.
    schema.py 에 정해둔 양식 그대로 돌려주도록 AI 호출 쪽을 다시 확인해줘.
    ```

    **시나리오 2: 조각 여러 장을 한 번에 못 보낸다**
    ```
    잘린 조각들을 한 번의 호출에 한꺼번에 보내야 해.
    조각마다 따로따로 AI를 부르면 사업부문 분리가 깨져.
    ```

## 2-3. 다른 공고도 같은 모듈로 돌려보기

메인 샘플(한화)로 모듈이 잘 돌아간다면, 이제 **공고 형태가 다른 3건**도 같은 모듈로 돌려봅니다. 자르기 모듈이 짧은 이미지를 괜히 자르지는 않는지, 추출 모듈이 포지션 하나짜리 공고도 잘 처리하는지 확인합니다.

!!! info "왜 이걸 확인하나요?"
    실제 업무에서는 공고 형태가 제각각입니다. 지금 만든 모듈이 **한화 공고에만 맞춤**이 아니라 **범용**이라는 확신이 필요해요. 짧은 이미지, 중간 길이, 매우 긴 이미지 — 세 가지 다른 형태를 한 번씩 돌려보면 감이 옵니다.

!!! quote "AI에게 이렇게 말해보세요 — 다른 공고 세 건 돌려보기"
    ```text
    방금 만든 모듈로 data/ 안의 다른 세 공고도 처리해줘.

    - data/codeit_2026.png       (짧은 이미지, 포지션 하나만 있을 가능성)
    - data/dongaoshuca_2026.png  (중간 길이 이미지)
    - data/hanmi_2026.jpg        (매우 긴 이미지)

    각각 output/<공고이름>/ 아래에 result.xlsx, result.html, chunks/ 가
    만들어졌는지 확인하고, 세 공고 각각 다음 항목을 한 줄씩 보고해줘.

    - 조각 수 (자르기가 필요 없었던 경우 "분할 없음" 표기)
    - 감지된 모집부문 수
    - 결과 파일 경로

    처리 중 이상한 점이 있으면 같이 알려줘.
    ```

!!! success "이런 결과가 보이면 정상입니다"
    - **codeit**: 분할 없이 원본 한 장 그대로 처리, 모집부문은 공고에 있는 만큼만
    - **dongaoshuca**: 1~2 조각으로 분할, 실제 공고의 모집부문 수만큼 감지
    - **hanmi**: 여러 조각으로 분할 (3~4장), 공고에 있는 모집부문 감지
    - 세 공고 모두 `result.xlsx`, `result.html` 이 잘 생성되고, 탐색기에서 열어봤을 때 내용이 자연스러움

!!! tip "이상하면?"
    - "codeit을 불필요하게 잘라버렸어" → "이미지 높이가 임계값 이하면 자르지 말고 원본 그대로 처리하는 조건이 제대로 들어있는지 image_slicer 를 다시 확인해줘"
    - "모집부문이 하나인데 억지로 여러 개로 쪼갰어" → "한 공고에 사업부문이 하나뿐이면 모집부문도 한 개만 만들어. 억지로 나누지 말고"
    - "특정 공고에서만 실패해" → "실패한 공고 하나를 집중해서 다시 돌려보고 에러 메시지와 함께 어디서 막혔는지 보여줘"
